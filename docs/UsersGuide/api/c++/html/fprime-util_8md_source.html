1 The `fprime-util` is a program designed to help developers run the standard development process by
2 automating many of the standard functions built into the build system. It is intended to remove some
3 of the inefficiencies using CMake while keeping the advantages of that build system.
4 
5 ## Basic Usage
6 ```
7 usage: fprime-util [-h] {generate,purge,hash-to-file,info,build,impl,check,coverage,install} ...
8 
9 F prime helper application.
10 
11 optional arguments:
12  -h, --help show this help message and exit
13 
14 subcommands:
15  F prime utility command line. Please run one of the commands. For help, run a command with the --help flag.
16 
17  {generate,purge,hash-to-file,info,build,impl,check,coverage,install}
18  generate Generate a build cache directory
19  purge Purge build cache directoriess
20  hash-to-file Converts F prime build hash to filename.
21  info Gets fprime-util contextual information.
22  build Build components, ports, and deployments in the specified directory
23  impl Generate implementation template files in the specified directory
24  check Run unit tests in the specified directory
25  coverage Generate unit test coverage reports in the specified directory
26  install Install the current deployment build artifacts in the specified directory
27 ```
28 
29 The `fprime-util` helper is driven by a series of subcommands listed above. Each perform one aspect
30 of the development process and are breifly described below. The following sections will go into
31 each command's usage in more detail.
32 
33 1. `generate`: generates build caches, one for development, and one with unit test flags enabled.
34  This is the required first step to using the F´ build system. This should be executed once int
35  the deployment directory before developing. It can be run again after the user runs `purge`.
36 2. `purge`: removes the build caches created with generate. This should be used when the build
37  system has not properly detected changes and need to be redone from the beginning.
38 3. `hash-to-file`: If F´ assertions are setup to output file hashes instead of file paths (i.e. when
39  `#define FW_ASSERT_LEVEL = FW_FILEID_ASSERT`), convert a given file hash to a full filepath.
40 4. `info`: Print information about location of build caches and `fprime-util` commands available in
41  the current directory.
42 5. `build`: builds the current directory. If in a component directory, it will compile the component
43  library. If in a deployment directory, it will build the binary.
44 6. `impl`: Autogenerate boilerplate for components and component unit tests. When creating a new
45  component, this command can generate a new skeleton implementation of the component from the
46  *Ai.xml file, saving significant development time. Providing the `--ut` flag will generate a unit
47  testing skeleton for the component.
48 7. `check`: Builds, if necessary, then runs unit tests in current directory. The `--all` runs all
49  unit tests known to a deployment. The `--leak` flag will check for memory leaks while running the
50  unit tests.
51 8. `coverage`: Similar to check, but calculates and generates unit test code coverage reports.
52 9. `install`: Builds, if necessary, then copies built deployment binaries into the `bin/` directory.
53 
54 Most `fprime-util` commands support a common set of flags that can be used to modify command
55 behavior:
56 
57 - `-j JOBS, --jobs JOB`: set the number of parallel jobs while compiling. Defaults to 1, but it's
58  recommended to set this to the number of cores on the machine + 1 to speed up compilation. Ex:
59  `fprime-util build -j 10` will compile a project with 10 simultaneous jobs.
60 - `--all`: run the supplied command in a global deployment-level scope, rather than in the current
61  directory. Ex: `fprime-util check` runs the unit tests in the current directory, `fprime-util
62  check --all` runs every available unit test.
63 - `--ut`: apply the supplied command to the unit tests, rather than the default release. Ex:
64  `fprime-util build --ut` will compile a project's unit tests.
65 - `-v, --verbose`: Produce verbose command output for debugging. Ex: `fprime-util build -v` will
66  build a project with extra debug output.
67 - `-p PATH, --path PATH`: `fprime-util` defaults to running in the current working directory. The
68  path flag allows users to override this behavior and manually set the directory `fprime-util`
69  runs in. Ex: `fprime-util check -p SigGen` run `check` in the SigGen subdirectory.
70 - `-d DEPLOY, --deploy DEPLOY`: By default, `fprime-util` starts in the current working directory
71  and starts moving up until it finds a F´ deployment. The deploy flag allows a user to override
72  this behavior to manually specify a deployment. Ex: `fprime-util build -d ~/fprime` will try to
73  build the `~/fprime` deployment.
74 
75 Additionally, there are some advanced flags that can be useful in some situations:
76 
77 - `--build-type {Release,Testing}`: Manually set the Cmake built type. This is usually automatically
78  determined by `fprime-util` and setting this can cause unexpected behavior.
79 - `--build-dir`: This sets the cmake build cache directory. This is usually automatically determined
80  by the deployment and build type and setting this can cause unexpected results.
81 
82 ## Setting Build Platform
83 
84 Different F´ deployments will target different operating systems, architectures, and hardware. To
85 support a variety of different platforms, `fprime-util` allows specifying a target platform to
86 run against. By default, `fprime-util` will target the `native` platform and build for the host
87 computer's platform.
88 
89 To cross compile for a different platform than that host computer, the platform positional argument
90 can be passed to `fprime-util`.
91 
92 For example, to cross compile for the Raspberry Pi, the `raspberrypi` platform can be used:
93 
94 ```
95 $ fprime-util generate raspberrypi
96 $ fprime-util build raspberrypi
97 ```
98 
99 If a deployment is targeting a specific platform, the deployment's `settings.ini` file can be used
100 to override the default platform from `native` to a platform of your choosing. See [the
101 `settings.ini` guide](./settings.md) for more details.
