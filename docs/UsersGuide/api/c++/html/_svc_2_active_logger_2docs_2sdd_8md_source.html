2 # Svc::ActiveLogger Component
3 
4 ## 1. Introduction
5 
6 The `Svc::ActiveLogger` component processes events from other components. The events are put in packets and sent to an
7 external component like the ground interface. The component provides event filtering capability such that events may
8 be turned off via ID or severity.

41 ### 3.2 Functional Description
42 
43 The `Svc::ActiveLogger` component provides an event logging function for the software. The framework autocoder allows
44 developers to specify a set of events in the component XML
45 (see [Events](https://nasa.github.io/fprime/UsersGuide/user/cmd-evt-chn-prm.html#Events)). For these components, the
46 autocoder will add an `Fw::Log` output port to send events in serialized form. The ActiveLogger receives these port
47 calls and provides commands to filter these events. The filtered events are sent to other components such as the ground
48 interface.
49 
50 **Note:** the Event ID value 0 is reserved for the logger.
51 
52 Should a FATAL severity event arrive, it is announced using a FATAL out port allow the system to respond when a FATAL
53 event is seen.
54 
55 #### 3.2.1 Filtering
56 
57 The `Svc::ActiveLogger` `LogRecv` input port handler filters events to lessen the load on the downstream components. The
58 filters can be set by severity and ID. By default, the DIAGNOSTIC events are filtered out since the number of DIAGNOSTIC
59 evens can be quite high. All defaults can be globally configured in `config/ActiveLoggerImplCfg.hpp`. Filters are
60 modified at runtime by the `SET_EVENT_FILTER` command.
61 
62 The component also allows filtering events by event ID. There is a configuration parameter that sets the number of IDs
63 that can be filtered. This allows operators to mute a particular event that might be flooding the downstream components.
64 These filters are modified at runtime by the `SET_ID_FILTER` command.
65 
66 FATAL events are never filtered, so they can be caught and broadcast to the system. Outgoing events are converted into
67 the F´ ground format and sent out using the `PktSend` port.
68 
69 
70 
71 #### 3.2.2 Fatal Announce
72 
73 When the `ActiveLogger` component receives a FATAL event, it calls the FatalAnnounce port. Another component that
74 handles the system response to FATALs (such as resetting the system) can connect to this port to be informed when a
75 FATAL has occurred.
