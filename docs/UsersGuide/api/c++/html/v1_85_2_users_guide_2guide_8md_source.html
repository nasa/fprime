1 # The Discerning User's Guide to F´
2 
3 F´ is an embedded systems framework developed at NASA's Jet Propulsion Laboratory for use in space based missions. Its
4 goal is simple: make development of flight-quality embedded software simple, easy, and fun. F´ reduces the complexity
5 and overhead to develop such systems.
6 
7 This guide is intended to expose the features of F´ and seek to allow users and developers to take advantage of all F´
8 has to offer. It is broken down into a getting started, users manual, best practices, developer documentation, and API
9 documentation sections.
10 
11 The Getting Started section points initiates to all the items needed to get up and running with F´. Since these are
12 often needed before a user searches for a user's guide, this section merely links to the stand-alone version of those
13 items.
14 
15 The User's Manual section helps users of this framework understand the concepts of F´ and build more complex projects
16 using the Framework. It is where users should go to understand more about what is possible in F´.
17 
18 The Best Practices section collects best practices and patterns many projects use while developing F´. Users who consult
19 this section will learn how to organize F´ projects and implementations to solve problems using standard and recommended
20 techniques.

79 ## Framework and Service Components
80 
81 F´ ships with a core framework and service code enabling the construction of F´ applications using standards.
82 Specifically, the service (Svc) components provide most of the basic functionality to command, and communicate with
83 an embedded system such as a spacecraft.
84 
85 ## Autocoder
86 
87 Autocoder automatically adds ports for registering commands, receiving
88 commands and reporting an execution status. F′ defines an Extensible
89 Markup Language (XML) schema that users can use to specify a model of a
90 FSW application. The model describes the application at a high level in
91 terms of the components, ports, and topologies of the F′ architecture.
92 The F′ autocoder translates the model into the C++ classes to greatly reduce the boiler plate code required to build
93 an embedded system.
94 
95 ## Operating System Abstraction Layer
96 
97 An operating system abstraction layer (OSAL) provides implementations of an application programming interface (API) for
98 real-time operating systems (RTOS) and non-real-time operating systems in a generic way such that the code using the
99 OSAL layer can easily swap to another OS without modification.
100 
101 The F′ framework OSAL includes C++ classes that provide abstractions of common operating systems features. These
102 features include threads, mutual exclusion locks, message queues, files, timers, and clocks.
103 
104 The open-source framework provides implementations of the operating systems layer for Linux and Mac OS. The Linux
105 implementation works in Windows using WSL. Some other Os are provided in extension libraries.
106 
107 ## Ground Data System (GDS)
108 
109 See: [F´ GDS](./gds/gds-introduction.md)
110 
111 The F′ user experience is intended to be an out-of-box ready to use ground data system (GDS) solution than can easily
112 run on Linux, Mas OSX, or Windows platforms without any mission specific tailoring. The GDS provides an end-user
113 graphical user interface (GUI) tool and an integration test application programming interface (API) to enable
114 integration testing and quick-look telemetry monitoring.
115 
116 Initially, the user defines the mission specific components with commands and telemetry, and then the dictionaries
117 are automatically generated from this design. These dictionaries are read into the GDS to provide the user experience.

120 ## F′ Utility Build Helper and CMake System
121 
122 F´ ships with a buoild system configured to build the F´ code. This build system is implemented using CMake and handles
123 the dependencies needed to run the Autocoder and assemble it with user written code. In order to support more standard
124 development patterns with F´, the `fprime-util` was built. This tool maps simple commands into the make system in order
125 to allow developer to follow set patterns easily.

127 ## Linux Drivers
128 
129 F´ ships with some basic driver implementations for Linux systems. These act as example driver components and allow
130 users choosing the Linux OS to run standard hardware like UART, I2C, SPI, etc.
