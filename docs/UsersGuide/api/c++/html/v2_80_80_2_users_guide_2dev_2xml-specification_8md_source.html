18 ## Serializable
19 
20 Serializables are the types that are passed between components in the
21 architecture and are used for ground data services such as commands,
22 telemetry, events, and parameters (see the component description in
23 Section 6.6.3 for what these are). A serializable type is a complex or
24 basic type that can be converted to a data buffer that contains the
25 values of an instance of the type. This data buffer can be passed around
26 the system (and to ground software) in a generic way and reconstituted
27 into the original type as needed. Basic C/C++ types like int and float
28 are supported automatically in the architecture, but the user is also
29 allowed to define arbitrary complex types that can be serialized. This
30 is done in one of two ways, either by hand-coding a class or allowing
31 the code generator to generate the class. These types can then be used
32 in XML specifications of ports and components.
33 
34 ### XML-Specified Serializable
35 
36 In most cases, a complex type is representable as a collection of simple
37 types. This is analogous to a C struct with data members. The code for
38 serializing and deserializing a struct is straightforward, so a code
39 generator is provided that will create the classes for a serializable
40 struct. The user specifies the members in XML, and the C++ class is
41 generated. The types of the members can be basic types or other
42 serializables, either XML or hand-coded. In order for the build system
43 to detect that a file contains the XML for a serializable type, the file
44 must follow the naming convention <SomeName>SerializableAi.xml. An
45 example of this can be found in
46 Autocoders/templates/ExampleSerializableAi.xml. Table 16 describes the
47 XML tags and attributes to use for the serializable.
48

76 #### Constraints
77 
78 ##### Ground Interfaces
79 
80 Serializables used for defining the interfaces for the ground system
81 must be fully specified in XML files. Members of the serializables
82 cannot be hand-coded types since the ground system analyzes the XML to
83 determine the types for displaying and archiving the data. (See Section

86 ### XML-Specified Enumeration
87 
88 #### Motivation
89 
90 As discussed in the previous section, you can define an enumeration
91 when specifying the type of a member of a Serializable type.
92 For example, you can create a file `SwitchSerializableAi.xml` containing
93 this specification:
94 
95 ```xml
96 <serializable name="Switch">
97  <members>
98  <member name="state">
99  <enum name="SwitchState" type="ENUM">
100  <item name="OFF" value="0">
101  <item name="ON" value="1">
102  </enum>
103  </member>
104  </members>
105 </serializable>
106 ```
107 
108 This file defines a Serializable type `Switch`
109 with one member `state`.
110 Its type is `SwitchState`, which is an enumeration with
111 enumerated constants `OFF` and `ON`.
112 
113 Alternatively, you can specify an enumeration *E* as a separate XML type.
114 Then you can do the following:
115 
116 1. Generate a C++ representation of *E* that you can include in C++
117 files and use on its own.
118 
119 2. Use the XML representation of *E* in Serializable XML types, in Array XML
120 types, in port arguments, in telemetry channels, and in event arguments.
121 
122 As an example, you can create a file `SwitchStateEnumAi.xml` that specifies an
123 XML enumeration type `SwitchState`
124 with enumerated constants `OFF` and `ON`, like this:
125 
126 ```xml
127 <enum name="SwitchState">
128  <item name="OFF" value="0"/>
129  <item name="ON" value="1"/>
130 </enum>
131 ```
132 
133 By running the code generator on this file, you can generate C++
134 files `SwitchStateEnumAc.hpp` and `SwitchStateEnumAc.cpp`
135 that define the C++ representation of the type.
136 Anywhere that you include `SwitchStateEnumAc.hpp` in your C++ code, you can
137 use the enumerated constants `SwitchState::OFF` and `SwitchState::ON`.
138 If you import `SwitchStateEnumAi.xml` into another XML definition,
139 then you can use the type `SwitchState` there.
140 
141 To use an XML enumeration type *E* in another XML definition *D*,
142 enclose the name of the file that defines *E* in an XML tag `import_enum_type`.
143 As an example, you can revise the `Switch` Serializable
144 definition shown above.
145 Instead of defining `SwitchState` as an inline enumeration, you can use the
146 `SwitchState` XML enumeration type as follows:
147 
148 ```xml
149 <serializable name="Switch">
150  <import_enum_type>SwitchStateEnumAi.xml</import_enum_type>
151  <members>
152  <member name="state" type="SwitchState"/>
153  </members>
154 </serializable>
155 ```
156 
157 Notice that the revised version (1) imports the enum type definition
158 from the file `SwitchStateEnumAi.xml` and (2) uses the named
159 type `SwitchState` as the type of member `state`.
160 
161 As another example, if you import the file `SwitchStateEnumAi.xml` into the
162 definition of a component *C*, then you can use the type `SwitchState` in the
163 telemetry dictionary for *C*.
164 When a value of type `SwitchState` is emitted as telemetry, the GDS
165 will display it symbolically as `OFF` or `ON`.
166 
167 #### Specification
168 
169 **File name:** An XML enumeration type *E* must
170 be defined in a file with the name *E* `EnumAi.xml`.
171 For example, the XML enumeration type `SwitchState`
172 must be defined in a file named `SwitchStateEnumAi.xml`.
173 
174 **Top-level structure:** An XML enumeration type is an XML node named `enum`
175 with attributes *enum_attributes* and children *enum_children*.
176 
177 `<enum` *enum_attributes* `>` *enum_children* `</enum>`
178 
179 **Enum attributes:** *enum_attributes* consists of the following:
180 
181 * An attribute `name` giving the name of the enumeration type.
182 
183 * An optional attribute `namespace` giving the enclosing namespace
184 of the enumeration type.
185 The namespace consists of one or more identifiers separated by `::`.
186 
187 If the attribute `namespace` is missing, then the type is
188 placed in the global namespace.
189 
190 _Examples:_ Here is an XML enumeration `E` in the global namespace:
191 
192 `<enum name="E">` ... `</enum>`
193 
194 Here is an XML enumeration `E` in the namespace `A::B`:
195 
196 `<enum name="E" namespace="A::B">` ... `</enum>`
197 
198 **Enum children:**
199 *enum_children* consists of the following, in any order:
200 
201 * An optional node `comment` containing comment text.
202 
203  `<comment>` *comment_text* `</comment>`
204 
205  The comment text becomes a comment in the generated C++ code.
206  It is attached to the C++ class that defines the enumeration.
207 
208 * One or more instances of *item_definition*, described below.
209 
210 **Item definition:**
211 *item_definition* defines an enumerated constant.
212 It is an XML node named `item` with the following attributes:
213 
214 * An attribute `name` giving the name of the enumerated
215 constant.
216 
217 * An optional attribute `value` assigning an integer value
218 to the enumerated constant.
219 If the `value` attribute is missing, then the value is assigned
220 in the ordinary way for C and C++ enumerations (i.e., zero for the first
221 constant, otherwise one more than the value assigned to
222 the previous constant).
223 
224 * An optional attribute `comment` giving comment text.
225 The text becomes a comment in the generated C++ code.
226 It is attached to the enumerated constant definition.
227 
228 _Examples:_ Here is an enumerated constant with a name only:
229 
230 `<item name="ON"/>`
231 
232 Here is an enumerated constant with a name, value, and comment:
233 
234 `<item name="OFF" value="0" comment="The off state"/>`

236 ### XML-Specified Array Type
237 
238 #### Motivation
239 
240 As discussed above, a member of a Serializable type can be an array of elements
241 of some other type.
242 For example, you can create a file `ACSTelemetrySerializable.xml` containing
243 this specification:
244 
245 ```xml
246 <serializable name="ACSTelemetry">
247  <members>
248  <member name="attitudeError" type="F32">
249  <member name="wheelSpeeds" type="U32" size="3">
250  </member>
251  </members>
252 </serializable>
253 ```
254 
255 This file defines a Serializable type `ACSTelemetry`
256 with the following members:
257 
258 * A member `attitudeError` of type `F32`.
259 
260 * A member `wheelSpeeds` whose type is an array of three values, each of type
261  `U32`.
262 
263 Alternatively, you can specify a named array type *A* in a separate
264 XML file.
265 Then you can do the following:
266 
267 1. Generate a C++ representation of *A* that you can include in C++
268 files and use on its own.
269 
270 2. Use the XML representation of *A* in Serializable XML types, in other Array
271 XML types, in port arguments, in telemetry channels, and in event arguments.
272 
273 As an example, you can create a file `WheelSpeedsArrayAi.xml` that specifies an
274 array of three `U32` values, like this:
275 
276 ```xml
277 <array name="WheelSpeeds">
278  <type>U32</type>
279  <size>3</size>
280  <format>%u</format>
281  <default>
282  <value>0</value>
283  <value>0</value>
284  <value>0</value>
285  </default>
286 </array>
287 ```
288 
289 By running the code generator on this file, you can generate C++
290 files `WheelSpeedsAc.hpp` and `WheelSpeedsAc.cpp`
291 that define a C++ class `WheelSpeeds` representing this type.
292 Anywhere that you include `WheelSpeedsEnumAc.hpp` in your C++ code, you can
293 use the class `WheelSpeeds`.
294 If you import `WheelSpeedsArrayAi.xml` into another XML definition,
295 then you can use the type `WheelSpeeds` there.
296 
297 To use an XML array type *A* in another XML definition *D*,
298 enclose the name of the file that defines *A* in an XML tag
299 `import_array_type`.
300 As an example, you can revise the definition of the Serializable type
301 `ACSTelemetry` as follows:
302 
303 ```xml
304 <serializable name="ACSTelemetry">
305  <import_array_type>WheelSpeedsArrayAi.xml</import_array_type>
306  <members>
307  <member name="attitudeError" type="F32"/>
308  <member name="wheelSpeeds" type="WheelSpeeds"/>
309  </members>
310 </serializable>
311 ```
312 
313 This specification defines an XML Serializable type with two members:
314 
315 1. Member `attitudeError` of type `F32`.
316 
317 1. Member `wheelSpeeds` of type `WheelSpeeds`.
318 
319 Notice that whereas before `wheelSpeeds` was directly declared as
320 an array of three `U32` values, here it is given type `WheelSpeeds`,
321 which is defined in a separate XML specification as an array
322 of three `U32` values.
323 
324 As another example, if you import file `WheelSpeedsArrayAi.xml` into the
325 definition of a component *C*, then you can use the type `WheelSpeeds` in the
326 telemetry dictionary for *C*.
327 When a value of type `WheelSpeeds` is emitted as telemetry, the GDS
328 will display it as an array of three values.
329 
330 #### Specification
331 
332 **File name:** An XML array type *A* must
333 be defined in a file with the name *A* `ArrayAi.xml`.
334 For example, the XML array type `WheelSpeeds`
335 must be defined in a file named `WheelSpeedsArrayAi.xml`.
336 
337 **Top-level structure:** An XML array type is an XML node named `array`
338 with attributes *array_attributes* and children *array_children*.
339 
340 `<array` *array_attributes* `>` *array_children* `</array>`
341 
342 **Array attributes:** *array_attributes* consists of the following:
343 
344 * An attribute `name` giving the name of the array type.
345 
346 * An optional attribute `namespace` giving the enclosing namespace
347 of the array type.
348 The namespace consists of one or more identifiers separated by `::`.
349 
350 If the attribute `namespace` is missing, then the type is
351 placed in the global namespace.
352 
353 _Examples:_ Here is an XML array `A` in the global namespace:
354 
355 `<array name="A">` ... `</array>`
356 
357 Here is an XML array `A` in the namespace `B::C`:
358 
359 `<array name="A" namespace="B::C">` ... `</array>`
360 
361 **Array children:**
362 *array_children* consists of the following, in any order:
363 
364 * An optional node `comment` containing comment text.
365 
366  `<comment>` *comment_text* `</comment>`
367 
368  The comment text becomes a comment in the generated C++ code.
369  It is attached to the C++ class that defines the array.
370 
371 * Zero or more of any of the following nodes:
372 
373  * `<include_header>` *header_file* `</include_header>` for including C++ header files
374  into the generated C++.
375 
376  * `<import_serializable_type>` *serializable_xml_file* `</import_serializable_type>`
377  for importing XML-specified serializable types.
378 
379  * `<import_enum_type>` *enum_xml_file* `</import_enum_type>` for importing
380  XML-specified enum types.
381 
382  * `<import_array_type>` *array_xml_file* `</import_array_type>` for
383  importing XML-specified array types.
384 
385 * A node `format` providing a single format string to be applied to each array element.
386 
387  `<format>` *format_string* `</format>`
388 
389  *format_string* must contain a single conversion specifier starting with `%`.
390  The conversion specifier must be legal both for C and C++ `printf` and for
391  Python, considering the array element type.
392  For example, if the array element type is `U32`, then `<format>%u</format>`
393  is a valid
394  format specifier. So is `<format>%u seconds</time>`.
395  `<format>%s</format>` is not a legal format specifier in this case, because
396  the string format `%s`
397  is not valid for type `U32`.
398 
399 * A node `type` consisting of attributes *type_attributes* and text *type*.
400 
401  `<type` *size_attribute_opt* `>` *type* `</type>`
402 
403  *size_attribute_opt* is an optional attribute `size` specifying a decimal
404  integer size.
405  The `size` attribute is valid only if the element type is `string`,
406  and it is required in this case.
407  It specifies the size of the string representation.
408 
409  *type* is text specifying the type of each array element.
410  It must be an F Prime built-in type such as `U32` or a named type.
411  A named type is (1) the name of an XML-specified type (Serializable, Enum, or Array)
412  or (2) the name of a C++ class included with `include_header`.
413 
414  *Examples:*
415 
416  * `<type>U32</type>` specifies the built-in type `U32`
417 
418  * `<type>T</type>` specifies the named type `T`.
419  `T` must be (1) included via `include_header` or (2) imported
420  via `import_serializable_type`, `import_enum_type`, or
421  `import_array_type`.
422 
423  * `<type size="40">string</type>` specifies a string
424  type of size 40.
425 
426 * A node `size` specifying the size of the array as a decimal integer.
427 
428  `<size>` *integer* `</size>`
429 
430 * A node `default` with one or more child nodes `value`.
431 
432  `<default>` _value_ ... `</default>`
433 
434  The format of _value_ is described below.
435  There must be one value for each element of the array.
436 
437 **Value:** The node `value` specifies a default value for an array
438 element.
439 The value must be text that, when inserted into the generated C++ code,
440 represents a default value of the correct type.
441 
442 *Examples:*
443 
444 * If the array element type is `U32`, then `0` is a correct default value.
445 
446 * Suppose the array element type is an array `A` of three U32 values.
447 In the generated C++ code, `A` is a class with a three-argument constructor,
448 and each argument initializes an array element.
449 Therefore `A(1, 2, 3)` is a correct default value.

452 ### Hand-coded Serializable
453 
454 A hand-coded serializable is useful in the case where the type is not
455 easily expressed as a collection of basic types or if the user is
456 employing an external library with types already defined. A user may
457 also wish to attach special processing functions to a class. The pattern
458 for writing C++ serializable classes is to declare a class that is
459 derived from the framework base class Fw::Serializable. That type can be
460 found in Fw/Types/Serializable.hpp. An example can be found in
461 Autocoders/templates/ExampleType.hpp. The method is as follows:
462 
463 1. Define the class. It should be derived from Fw::Serializable.
464 
465 1. Declare the two base class pure virtual functions: serialize() and
466  deserialize(). These functions provide a buffer as a destination for
467  the serialized form of the data in the type.
468 
469 1. Implement the functions. The buffer base class that is the argument
470  to those functions provides a number of helper functions for
471  serializing and deserializing the basic types in the table above.
472  Those functions are specified in the SerializeBufferBase class type
473  in the same header file. For each member that the developer wishes
474  to save, call the serialization functions. The members can be
475  serialized in any order. In the worst case, a raw buffer version is
476  provided for just doing a bulk copy of the data if serializing
477  individual members is not feasible. The deserialization function
478  should deserialize the data in the order it was serialized.
479 
480 1. Add an enumeration with a single member named SERIALIZED\_SIZE. The
481  value of that member should be the sum of the sizes of all the
482  members. It can be done with the sizeof() function that is available
483  to all C/C++ compilers. Optionally, a type identifier can be added
484  so that a sanity check can be done when the data are deserialized.
485  That type should be serialized, and then checked against the
486  enumeration when it is deserialized.
487 
488 1. Implement copy constructors and equal operators. If the type is
489  passed by value, the developer should write these functions if the
490  default is not sufficient.
491 
492 1. Implement any custom features.
493 
494 Once the class is completed, it can be included in port definitions.
495 
496 #### Constraints
497 
498 ##### Ground Interfaces
499 
500 Hand-coded user types cannot be used in XML definitions for types that
501 will be sent to and from the ground system, since the ground system
502 requires all type definitions to be in XML. If a developer wants to use
503 a type that has special functions in a ground interface, it can be
504 defined with the members only in XML and generated by the code
505 generator. The developer can then define a derived class with the
506 behavior.

508 ## Port
509 
510 Ports are the interfaces between components. Components invoke
511 functionality in other components by invoking methods on their output
512 ports instead of invoking the components directly. Input ports invoke
513 methods defined by the components and registered at run time. Ports are
514 fully specified in XML files. There is no developer-written code, but
515 the XML specifications can include argument types defined by the user as
516 long as they are serializable as described in Section 6.6.1.2. The
517 process of defining a port is to specify the arguments to the method in
518 the XML file. In order for the build system to detect that a file
519 contains the XML for a port type, the file must follow the naming
520 convention <SomeName>PortAi.xml. An example of this can be found in
521 Autocoders/templates/ExamplePortAi.xml. Table 17 describes the XML tags
522 and attributes.
523

525 specification.
526

529 | interface | | The outermost tag that indicates a port is being defined. |
530 | interface | namespace | The C++ namespace for the port class (optional). |
531 | interface | name | The class name for the port. |
532 | comment | | Used for a comment describing the port. Is placed as a Doxygen-compatible tag in the class declaration. |
533 | args | | Optional. Starts the region of the declaration where port arguments are specified. |
534 | arg | | Defines an argument to the port method. |
535 | arg | type | The type of the argument. Should be a built-in type, ENUM, string, an XML-specified type, or a user-written serializable type. A string type should be used if a text string is the argument. If the type is serial, the port is an untyped serial port that can be connected to any typed port for the purpose of serializing the call. 537 | arg | size | Specifies the size of the argument if it is of type string. |
538 | arg | pass\_by | Optional. Specifies if the argument should be passed by reference or pointer. Default is to pass by value. Values can be VALUE, POINTER, or REFERENCE. 540 | enum | name | Enumeration type name. |

543 | item | value | Assigns a value to the enumeration member. Member values in the enumeration follow C enumeration rules if not specified. |
544 | item | comment | A comment about the member. Becomes a Doxygen tag. |
545 | return | | Optional. Specifies that the method will return a value. |
546 | return | type | Specifies the type of the return. Can be a built-in type. |
547 | return | pass\_by | Specifies whether the argument should be passed by value or by reference or pointer. |

553 ### Constraints
554 
555 The following constraints are on port XML definitions.
556 
557 #### Serialization
558 
559 Port calls are serialized for various reasons by components, such as for
560 copying data to put on a message queue. A port call cannot be serialized
561 if it contains return values, since the serialized arguments are passed
562 only to the connected port or component and no return data are provided.
563 
564 ## Components
565 
566 Components are the places in the architecture where all the behavior
567 resides. Incoming port calls deliver data and actions to be acted on,
568 and output ports are used to invoke functions outside the component. In
569 addition, the components define the interfaces used to communicate with
570 ground software. Components can be used for a variety of purposes such
571 as hardware drivers, state machines, device managers, and data
572 management. The process of defining a component is to specify the ports
573 and ground data interfaces in XML. The developer then implements a
574 derived class that inherits from the base class and implements the
575 interfaces as methods in the class (see Section 6.7). In order for the
576 build system to detect that a file contains the XML for a component
577 type, the file must follow the naming convention
578 <SomeName>ComponentAi.xml. An example of this can be found in
579 Autocoders/templates/ExampleComponentAi.xml. Table 18 describes the XML
580 tags and attributes.
581 
582 **Table 18.** Component XML
583 specification.
584 
585 | Tag | Attribute | Description | |
586 | -------------------------- | --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
587 | component | | The outermost tag that indicates a component is being defined. | |

709 ### Constraints
710 
711 The following constraints are on components.
712 
713 #### Passive Components
714 
715 Passive components cannot have asynchronous input ports or commands,
716 since there is no queue for messaging.
717 
718 #### Queued Components
719 
720 Queued components must have at least one synchronous or guarded input
721 port or synchronous command so that a calling thread can use the port to
722 retrieve port call messages from the message queue.
723 
724 Queued components must have at least one asynchronous input port,
725 command, or internal interface, or there would be no purpose in making
726 the component queued.
727 
728 #### Active Components
729 
730 Active components must have at least one asynchronous input port,
731 command, or internal interface, or there would be no purpose in making
732 the component active or have a message queue.
733 
734 #### Command and Telemetry Interfaces
735 
736 The component XML is parsed by the command and telemetry system in order
737 to construct the data storage and display application. For this reason,
738 only the built-in types or types represented in XML can be
739 used.
740 
741 ## Command/Telemetry Ports
742 
743 When declaring commands, telemetry, events, and parameters for a
744 component, the code generator automatically creates the correct set of
745 ports for those interfaces. Those ports are based on normal XML
746 definitions of ports, so those port types can be used as normal ports in
747 other components. As an example, if a component needs to implement
748 commands, the XML would declare those commands as shown in Table 1.
749 Another component needs to act as a command dispatcher to send commands
750 to that component, so the dispatcher component would add an output port
751 of the command type. The dispatcher is handling commands as a generic
752 port invocation rather than deserializing the command arguments as the
753 generated code does in the component that declares the commands. This
754 allows writing components to process commands and telemetry generically
755 without having to treat command and telemetry ports as special cases.
756 The port definition XML files can be included in component XML
757 definitions in the same manner as other ports. Table 19 provides a list
758 of the types, file names, and descriptions of the ports used for the
759 command and telemetry ports.
